## 需求到设计六步流程 × ECC Skills 映射
collapsed:: true
	- ### Step 1: 需求澄清（Quest 模式）
		- 目标：用 5W1H / 用户故事梳理需求，多方对齐
		  | ECC Skill / Agent        | 作用                                    |
		  | ------------------------ | ------------------------------------- |
		  | **`search-first`**       | **研究优先工作流** — 编码前先进行多源研究，确保需求基于事实而非假设 |
		  | **`market-research`**    | 市场、竞品、投资者研究，带来源归因 — 适合产品需求调研阶段        |
		  | **`deep-research`**      | 多源研究 + 综合 + 来源归因 — 深度需求背景调查           |
		  | **`product-capability`** | 将产品目标转化为有范围的能力地图 — 需求范围界定             |
		  | **`planner` Agent**      | 功能实现规划 — 将模糊需求拆解为可执行计划                |
		  | **`/ecc:plan` Command**  | 生成实现蓝图 — 对应 Quest 模式的多轮澄清             |
		- 使用方式：
		  ```
		  /ecc:plan "用户认证系统需求澄清"
		  → planner agent 先拆解任务，再逐模块确认边界
		  ```
	- ### Step 2: 生成 PRD（SDD 模式 / SKILL 模式）
	  collapsed:: true
		- 目标：生成结构化的需求文档 / 产品需求文档
		  | ECC Skill / Agent          | 作用                                        |
		  | -------------------------- | ----------------------------------------- |
		  | **`article-writing`**      | 长文写作（基于提供的语料和风格参考）— 可用于撰写 PRD 文档          |
		  | **`investor-materials`**   | pitch deck、备忘录、财务模型 — 若需求涉及商业化可辅助商业需求文档   |
		  | **`brand-voice`**          | 从真实内容中提取写作风格 — 确保 PRD 语气一致                |
		  | **`content-engine`**       | 多平台内容工作流 — 若需求涉及内容产品                      |
		  | **`documentation-lookup`** | 通过 Context7 MCP 获取最新库和框架文档 — 确保 PRD 技术可行性 |
		  | **`/update-docs` Command** | 文档同步更新                                    |
		- **使用方式**：
		  结合 `search-first` 的研究结果，用 `article-writing` 生成结构化的 PRD 文档，确保技术可行性通过 `documentation-lookup` 验证。
	- ### Step 3: 生成设计文档（SDD → PDD）
	  collapsed:: true
		- 目标：系统架构、数据库设计、API 设计、接口契约
		  | ECC Skill / Agent         | 作用                                              |
		  | ------------------------- | ----------------------------------------------- |
		  | **`architect` Agent**     | **系统设计决策** — 核心架构设计 Agent                       |
		  | **`api-design`**          | REST API 设计模式、分页、错误响应 — API 设计规范                |
		  | **`backend-patterns`**    | API、数据库、缓存模式 — 后端架构设计                           |
		  | **`frontend-patterns`**   | React/Next.js 模式 — 前端架构设计                       |
		  | **`database-migrations`** | 迁移模式（Prisma、Drizzle、Django、Go）— 数据库设计           |
		  | **`postgres-patterns`**   | PostgreSQL 优化模式 — 数据库详细设计                       |
		  | **`docker-patterns`**     | Docker Compose、网络、卷、容器安全 — 部署架构                 |
		  | **`deployment-patterns`** | CI/CD、健康检查、回滚 — 运维架构设计                          |
		  | **`mcp-server-patterns`** | 用 Node/TypeScript SDK 构建 MCP 服务器 — 若涉及 AI 工具链集成 |
		  | **`security-review`**     | 综合安全清单 — 设计阶段安全评审                               |
		- 使用方式：
		  ```
		  /ecc:plan "设计用户认证系统架构"
		  → architect agent 输出系统架构决策
		  → 调用 api-design + backend-patterns 细化 API 和数据库设计
		  → security-review 进行安全评审
		  ```
	- ### Step 4: 任务拆分（/plan 模式）
	  collapsed:: true
		- 目标：按模块/Phase 拆分，生成 task.md，接入 Jira
		  | ECC Skill / Agent             | 作用                                         |
		  | ----------------------------- | ------------------------------------------ |
		  | **`planner` Agent**           | **功能实现规划** — 核心任务拆解 Agent                  |
		  | **`/ecc:plan` Command**       | 生成实现计划 — 对应 /plan 模式                       |
		  | **`dmux-workflows`**          | 多 Agent 编排（使用 tmux pane manager）— 复杂任务并行拆解 |
		  | **`autonomous-loops`**        | 自主循环模式：顺序管道、PR 循环、DAG 编排 — 自动化任务流          |
		  | **`/multi-plan` Command**     | 多模型协作规划 — 大型项目多 Agent 并行规划                 |
		  | **`/multi-workflow` Command** | 完整多模型开发工作流                                 |
		- 使用方式：
		  ```
		  /multi-plan "用户认证系统开发"
		  → 多模型协作拆解为：数据库模块、API 模块、前端模块、测试模块
		  → 生成 task.md 并标记优先级
		  ```
	- ### Step 5: 原型验证（Figma / 墨刀）
	  collapsed:: true
		- 目标：前端原型确认，技术方案评审
		  | ECC Skill / Agent             | 作用                                             |
		  | ----------------------------- | ---------------------------------------------- |
		  | **`frontend-slides`**         | HTML 演示文稿、PPTX 转换、视觉风格探索 — 快速原型展示              |
		  | **`frontend-patterns`**       | React/Next.js 模式 — 前端技术方案验证                    |
		  | **`e2e-testing`**             | Playwright E2E 测试 + Page Object Model — 原型功能验证 |
		  | **`/code-review` Command**    | 代码评审 — 技术方案评审                                  |
		  | **`code-reviewer` Agent**     | 质量、安全、可维护性评审                                   |
		  | **`security-reviewer` Agent** | 漏洞分析 — 原型安全评审                                  |
		- **使用方式**：
		  用 `frontend-slides` 生成可交互的 HTML 原型演示，结合 `e2e-testing` 编写 Playwright 测试验证核心用户流程。
	- ### Step 6: 开发 & 迭代（需求变更管理）
	  collapsed:: true
		- 目标：需求变更时，走变更流程，更新文档并评估影响
		  | ECC Skill / Agent              | 作用                                 |
		  | ------------------------------ | ---------------------------------- |
		  | **`verification-loop`**        | 持续验证：构建、测试、Lint、类型检查、安全 — 变更后的质量门禁 |
		  | **`/quality-gate` Command**    | 验证门禁检查                             |
		  | **`/checkpoint` Command**      | 保存验证状态 — 变更前打检查点                   |
		  | **`/evolve` Command**          | 将学习到的模式聚类为 Skills — 迭代优化           |
		  | **`continuous-learning-v2`**   | 基于本能的学习（带置信度评分）— 从变更中提取模式          |
		  | **`ai-regression-testing`**    | AI 回归测试 — 变更影响评估                   |
		  | **`/update-docs` Command**     | 文档同步 — 变更后更新设计文档                   |
		  | **`/update-codemaps` Command** | 更新代码地图 — 变更后架构可视化                  |
		  | **`doc-updater` Agent**        | 文档同步 Agent                         |
		- 使用方式：
		  ```
		  /checkpoint                    # 变更前保存状态
		  # 执行需求变更...
		  /quality-gate                  # 运行质量门禁
		  /update-docs                   # 同步更新文档
		  /evolve                        # 提取新模式到 Skills
		  ```
- ## OpenSpec使用
	- ### 安装
		- ```
		  # AI端
		  参考 https://github.com/Fission-AI/OpenSpec 安装OpenSpec
		  ```
		- ```
		  # 或者命令
		  npm install -g @fission-ai/openspec@latest
		  # 中文補丁
		  npm install -g git+https://github.com/Boris-code/openspec-cn.git
		  ```
	- ### AI端命令
		- ```
		  /opsx:propose → /opsx:explore → /opsx:apply → /opsx:sync → /opsx:archive
		                ↑
		           發現需求不夠清晰，先探索再繼續
		  ```
		- | 命令                     | 作用                                                                                           |
		  | ---------------------- | -------------------------------------------------------------------------------------------- |
		  | `/opsx:explore "你的想法"` | 深入思考想法，探究问题，用戶只有大致想法，不確定具體怎麼做，需要 AI 幫忙梳理和探索可行方案 |
		  | `/opsx:propose "你的想法"` | **提出需求變更**。AI 会自动创建变更目录，包含 `proposal.md`（提案）、`specs/`（需求规格）、`design.md`（技术方案）、`tasks.md`（任务清单） |
		  | `/opsx:sync`          | **同步**狀態，確保雙方一致                                                        |
		  | `/opsx:apply`          | **执行实现**。AI 按照 `tasks.md` 逐条完成任务并修改代码                                                        |
		  | `/opsx:archive`        | **归档变更**。将完成的变更移动到 `openspec/changes/archive/`，并把 delta specs 合并到主规格中                        |
	- ### 终端命令
		- `openspec init`：初始化openspec配置

		- `/opsx:propose xxx`：创建需求变更
